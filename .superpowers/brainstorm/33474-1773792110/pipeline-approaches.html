3 Pipeline Architecture Approaches

Where does the score computation actually run? This is the heart of "very strong pipeline."

A

PostgreSQL-Native Pipeline

Score engines live INSIDE the database as SQL functions + triggers. Every INSERT fires score recomputation. Supabase Realtime pushes changes to dashboard.

INSERT health_sample → trigger → compute_recovery_score() → UPDATE user_scores → Realtime push → dashboard updates

Pros

  • Zero external services — everything in Supabase
  • Atomic: score always consistent with data
  • $0 cost, no compute limits
  • Fastest possible latency (same machine)
  • Works 24/7 without any server running

Cons

  • Complex SQL for score algorithms
  • Harder to debug than TypeScript/Python
  • 14-day baseline windows = heavy queries on trigger
  • Limited to what PostgreSQL can compute
B

Edge Functions Pipeline

Supabase Edge Functions (Deno/TypeScript) handle ingestion + score computation. Triggered by database webhooks or direct HTTP calls from the app.

App → POST /functions/ingest-health → validate + store → call compute-scores → UPDATE user_scores → Realtime push

Pros

  • TypeScript — same language as frontend
  • Easy to test and debug locally
  • Complex algorithms are readable
  • 500K free invocations/mo
  • Serverless — no server to manage

Cons

  • 50ms cold start on first call
  • 2-min max execution time per invocation
  • Extra network hop (function → DB → Realtime)
  • Free tier limit could matter at scale
C

Hybrid: Triggers + Edge Functions

Simple, fast operations in PostgreSQL triggers. Complex score algorithms in Edge Functions triggered by DB webhooks. Best of both worlds.

INSERT health_sample → trigger: validate + tag + notify → Edge Function: compute complex scores → UPDATE user_scores → Realtime push

Pros

  • Fast path: simple ops in DB (validation, tagging, aggregation)
  • Smart path: complex algorithms in TypeScript (TRIMP, AHEI, baselines)
  • Each layer does what it's best at
  • Testable TypeScript for business logic
  • Atomic DB operations for data integrity

Cons

  • Two systems to maintain
  • Slightly more complex debugging

My Recommendation: Option C (Hybrid)

Here's exactly what goes where:

PostgreSQL Triggers (instant, in-DB)
  • Data validation + deduplication
  • Daily aggregate rollups (avg HR, total steps, etc.)
  • Weight rate-of-change calculation
  • Simple threshold alerts (HR > 120, SpO2 < 94)
  • Updated_at timestamps on score tables
  • Notify channels for Realtime
Edge Functions (complex, TypeScript)
  • Recovery Score (14-day baseline, multi-input weighted)
  • Strain Score (TRIMP, logarithmic scaling, HR zones)
  • Sleep Score (stages + HR dip + efficiency + debt)
  • Nutrition Score (AHEI calculation, food quality)
  • Energy Bank (composite of 4 scores)
  • Trend detection (anomaly detection, slope analysis)
  • Cardio Load (ATL vs CTL, 7 status categories)
  • Smart alarm computation

The Pipeline Flow

1. App sends health data → Edge Function: ingest-health (validate, normalize, store)
2. INSERT into health_samples → PG trigger: update daily_aggregates
3. PG trigger → pg_net: call Edge Function compute-scores
4. Edge Function reads 14-day window → computes all scores → UPSERTs user_scores
5. user_scores UPDATE → Supabase Realtime → dashboard auto-updates
6. If anomaly detected → push notification via Edge Function
